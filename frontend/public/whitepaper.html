○ Section 1

Abstract

The Adaptive University Intelligence Platform (AUIP) is an enterprise-grade, cloud-native operating system for higher education institutions. It replaces fragmented legacy workflows — manual Excel-based placement tracking, self-service student sign-up portals, and institutional data silos — with a single, intelligence-driven ecosystem.

AUIP is built on four foundational pillars: institution-provisioned identity, zero-trust security, strict PostgreSQL schema-based multi-tenancy, and an AI Governance Brain that provides placement readiness scoring and personalized interventions.

The platform is currently in active development under Sprint 2 (Student Pre-Seeding & Activation), with core authentication, multi-tenancy, and institutional onboarding completed in Sprint 1.

💡 Core Philosophy: Identity is institution-provisioned, not user-created. Students do not "create accounts" — they confirm identities that have been pre-seeded by their institution. This ensures zero wasted resources on dormant accounts and gives institutions complete data sovereignty over their student population.

○ Section 3

The AUIP Identity Model

3.1 Institution-Provisioned Identity

AUIP fundamentally rethinks how student accounts are created. The platform operates on a pre-seeded identity model: universities upload their student roster (via CSV or manual entry), and the system creates internal identity records without allocating any user account resources.

Each student is assigned a unique STU_REF (e.g., STU_2025_0142) — the canonical identity anchor. Email is used only for communication and verification, not as an identity key. This separation means:

  • Email changes do not break existing identity or session history
  • Multiple communication channels can be added without re-registration
  • Institutions retain full control over who can activate an account
  • Zero resources are consumed for students who never activate

3.2 Student Lifecycle State Machine

Every student identity passes through exactly 4 states, with no ability to skip or reverse (except ACTIVE ↔ INACTIVE for admin deactivation):

SEEDED → INVITED → VERIFIED → ACTIVE │ │ │ │ │ │ │ └── Full account: User record, │ │ │ JWT sessions, dashboard access, │ │ │ placement eligibility engine. │ │ │ │ │ └── OTP or activation link verified. │ │ Password set. User account created. │ │ │ └── Signed activation email sent (TimestampSigner token, │ 24h expiry, single-use, invalidates previous tokens). │ └── Admin uploads CSV. Internal identity record created. No User account. No resources allocated.
🌱

SEEDED

Institution uploads roster. STU_REF assigned. No account exists. No resources consumed. Pure identity record.

✉️

INVITED

Admin triggers activation batch. Signed token emailed. Student cannot proceed without institutional invite.

✅

VERIFIED

Student clicked link, verified OTP, set password. Identity confirmed.

🚀

ACTIVE

Full account provisioned. JWT sessions, Quantum Shield cookies, dashboard access, placement eligibility.

4.2 Multi-Tenant PostgreSQL Isolation

Every approved institution receives its own dedicated PostgreSQL schema (inst_<slug>). This is not row-level isolation — it is schema-level physical isolation. Queries running in the context of inst_mit are structurally incapable of reading data from inst_iit_delhi.

PostgreSQL Database
├── public schema
│   ├── User                  (platform-wide identity)
│   ├── Institution           (registration + approval records)
│   ├── LoginSession          (device-bound sessions)
│   ├── BlacklistedAccessToken
│   ├── RememberedDevice
│   └── PasswordResetRequest

├── inst_mit schema
│   ├── CoreStudent           (academic full records, secured)

🔐 Security Guarantee: Cross-tenant queries are physically impossible. PostgreSQL's search_path is set to the institution's schema at the connection level. No application-level filtering can accidentally expose another institution's data.
○ Section 5

Security Model — Quantum Shield

AUIP's security architecture is built around defense in depth: six independent layers, each one capable of catching what the previous layer missed.

5.1 Quantum Shield — Quad-Segment Cookie Fragmentation

Traditional applications store the refresh JWT as a single HttpOnly cookie. If that cookie is ever stolen via XSS, MITM, or browser exploit, the attacker gains complete session access. AUIP eliminates this attack vector entirely.

The refresh JWT (header.payload.signature) is split into 4 separate cookies with different security properties. All 4 segments are required to reconstruct a valid token. Stealing any subset is cryptographically useless.

SegmentCookie NameContentsHttpOnlySameSite
T (TTL)_auip_sh_tHeader + first half of payload❌ NoLax
ID (Session)_auip_sh_idOpaque LoginSession ID✅ YesStrict
P (Payload)_auip_sh_pSecond half of payload✅ YesStrict
S (Signature)_auip_sh_sJWT cryptographic signature✅ YesStrict
🧱 An XSS attack can only steal Segment T — the only JS-accessible cookie, containing only TTL metadata. Segments ID, P, and S are HttpOnly. Without all four segments, an attacker cannot reconstruct any valid JWT, making session hijacking via XSS structurally impossible.

5.2 SafeJWT Triple-Check Authentication

Every API request is validated against three independent layers:

Layer 1: JWT Signature + Expiry └── SimpleJWT validates cryptographic signature and exp claim. Layer 2: Database Session Whitelist └── LoginSession with matching JTI must be active in the DB. If the session was force-logged-out, the token is useless even if it hasn't expired. Layer 3: Device Fingerprint └── SHA256(IP + User-Agent + session-unique salt) must match the hash stored at session creation time. Fingerprint mismatch → session killed + JTI blacklisted + WebSocket force_logout to all user devices.

5.3 Defensive Security Stack

🔒

Brute-Force Protection

After 5 failures per (identifier, IP): account locked for 5 min. Redis-backed, per-identifier tracking with configurable thresholds.

🌐

Global IP Lockout

After 5 platform-wide failures from same IP: 10-min block. Email incident report auto-sent to Super Admin with IP, target, user-agent, and timestamp.

🔑

HMAC Token Hashing

All stored tokens (OTP, reset, access) are HMAC-SHA256 hashed. Format: key_id$hmac_hex. Key rotation supported at runtime without user disruption.

🤖

Cloudflare Turnstile

Bot protection on all public endpoints. Configurable ON/OFF per environment. Widget resets on submission failure to prevent token reuse.

🔄

Silent Token Rotation

Access tokens auto-renewed 15 seconds before expiry. New tokens delivered via response header. Old refresh JTI immediately blacklisted.

🛡️

Content Security Policy

Strict CSP headers via middleware. default-src: 'self'. Only Cloudflare's challenge domain whitelisted for iframes/scripts.

AI Governance Brain

The Governance Brain is AUIP's intelligence layer — an advisory engine that aggregates student performance data across academics, mock tests, interviews, and projects to produce a Placement Readiness Score (0–100) for every student, updated daily.

🧠 Design principle: AI is used only to assist, never to override institutional authority. Every AI recommendation is labelled as such and can be dismissed or acted upon by the TPO. No student is automatically blocked, accepted, or penalised based on an AI score.

6.2 At-Risk Detection & Intervention

Students with readiness scores below 40 are automatically flagged as "at-risk" in the TPO dashboard. For each flagged student, the Governance Brain surfaces:

  • Specific weak areas (e.g., "Aptitude: 28th percentile, Verbal: 15th percentile")
  • Recommended mock assignments targeting those weak areas (max 2/week to avoid cognitive overload)
  • Predicted placement probability based on historical cohort data
  • Suggested interventions (mentoring, additional mocks, resume workshop)

6.3 Intelligence Service (ML Layer)

The Intelligence Service (Sprint 5) sits beneath the Governance Brain and powers its predictions with trained models:

📈

ML model predicting per-student placement probability based on cohort history.

🔍

Anomaly Detection

Autoencoders flag unusual performance patterns (e.g., sudden CGPA drop, mock abandonment spikes).

📄

NLP Resume Parsing

Extracts skills, projects, and experience from uploaded PDFs for ATS scoring and eligibility matching.

Placement Management Engine

AUIP replaces Excel-based placement workflows with a dynamic, rule-engine-driven system. TPOs create drives with compound eligibility criteria; the engine evaluates all students automatically and in real-time.

7.1 Dynamic Eligibility Engine

Criteria are defined using structured AND/OR/nested logic:

Example: Google SWE Drive — Eligibility Rule
{
  "operator": "AND",
  "conditions": [
    { "field": "cgpa",        "op": "gte", "value": 7.5 },
    { "field": "tenth_pct",   "op": "gte", "value": 60  },
    {
      "operator": "OR",
      "conditions": [
        { "field": "branch", "op": "in", "value": ["CS", "IT", "ECE"] },
        { "field": "twelfth_pct", "op": "gte", "value": 75 }
      ]
    }
  ]
}

7.2 One-Student-One-Job Rule

The system enforces hard constraints at the application level. Once a student's placement is marked as successful in any drive, they cannot apply to further active drives. This mirrors real-world campus placement policy and prevents unfair multi-offer scenarios.

○ Section 9

Platform Policies

9.1 Data Privacy & Sovereignty

  • Each institution's data is physically isolated in its own PostgreSQL schema. Cross-tenant access is structurally impossible — not merely policy-enforced.
  • Student academic records (CGPA, 10th/12th marks, semester grades) are stored in a secured tenant schema, visible only to the institution's authorised administrators and the student themselves.
  • The platform does not share, sell, or process student data for any purpose outside the institution's explicit operational scope.
  • AUIP is architected for GDPR compliance: all personal data fields are identifiable, exportable, and deletable on request.

9.2 Acceptable Use

  • The platform is available exclusively to institutions approved by the Super Admin. Unapproved access attempts are blocked at the registration level (Turnstile) and the authentication level (RBAC).
  • Students may not register without an institutional invitation. Self-registration creates no account and no identity record.
  • The AI Governance Brain's recommendations are advisory only. No automated decisions are made without human review by a TPO or administrator.
  • Anti-cheat mechanisms (tab-switch detection, fullscreen enforcement) are disclosed to students before any mock test begins.

9.3 Security Responsibilities

  • Institutions are responsible for the accuracy of pre-seeded student data. AUIP validates format and deduplication but cannot verify ground-truth academic records.
  • Administrators must not share credentials. Each role (Super Admin, Institution Admin, Faculty, Student) has strictly scoped permissions with no privilege escalation path.
  • The platform maintains detailed LoginSession records. Suspicious sessions (IP change, device fingerprint mismatch) are automatically deactivated and reported.
  • Password reset tokens are single-use and time-limited (24-hour default). Unused tokens are invalidated when a new reset request is made.

9.4 Uptime & SLA

  • Target uptime: 99.9% (excluding scheduled maintenance windows)
  • Redis-backed caching for all public API endpoints (5-min TTL). Cache invalidation is automatic on content updates.
  • WebSocket connections are managed by Django Channels with automatic reconnection on the frontend.
  • Database read replicas are planned for production deployment (Sprint 9+) to ensure read scalability during peak placement seasons.
○ Section 10

Conclusion

AUIP is not a new face on an old workflow. It is a ground-up rethinking of how universities manage student identity, placement, and academic governance. By making identity institution-provisioned, security structurally enforced (not just policy-enforced), and decision-making AI-assisted (not AI-automated), AUIP provides a platform that is simultaneously more secure, more efficient, and more equitable than any existing system.

The core authentication and multi-tenancy infrastructure is complete and battle-tested. The platform is now actively building the student lifecycle, placement, and AI governance layers — all on a foundation designed to carry them reliably at institutional scale.
